Circuit Breaker​

개요​

  • 특정 서버에 실행하는 요청의 실패 빈도가 너무 높아진다면 잠시 요청을 차단한다.
  • 이를 통해 서버의 부하를 줄이고 회복할 시간을 준다.
  • 주로 스파이크 트래픽 등 순간적으로 폭증하는 트래픽 대응에 유효하다.

주의점​

  • 클라이언트 입장에서 서킷브레이커의 fallback 동작으로 재요청을 보내는 경우 서버에서는 이미 큐에 넣고 실행중인 동작일 수 있다.
  • 이런 경우 중복실행의 위험이 있으며 서킷브레이커 동작 대상이 되는 API는 멱등하게 설계하는 것이 중요하다.
  • 아래 예제코드와 같이 서킷브레이커 동작 대상 과 그렇지 않은 예외를 잘 구분한다.
@Override
@CircuitBreaker(name = MAIL_SERVICE_CB, fallbackMethod = "sendMailFallback")
public String sendMail(String email) {
    log.debug("[Circuit Breaker: {}] Attempting to send mail for email: {}", MAIL_SERVICE_CB, email);
    EmailRequest request = new EmailRequest(email);

    // WebClient 호출 및 결과 처리
    // .block()을 사용하므로, 전체 메서드는 동기적으로 동작합니다.
    // 완전한 비동기 처리를 원한다면 Mono<String>을 반환하도록 수정해야 합니다.
    return webClient.post()
            .uri(MAIL_SEND_URI) // 상수 사용
            .bodyValue(request)
            .retrieve()
            // --- 5xx 서버 에러 처리 ---
            // onStatus를 사용하여 특정 상태 코드에 대한 처리를 명확히 함
            .onStatus(status -> status.is5xxServerError(), clientResponse ->
                    clientResponse.bodyToMono(String.class)
                            .flatMap(errorBody -> {
                                // 5xx 에러는 별도의 예외(MailSendException)로 처리 (서킷 기록 안 됨 명시)
                                // 장애 응답을 서버가 제때 준 것 이므로 서킷브레이커 동작 대상이 아님
                                log.warn("[CB: {}] Mail service returned 5xx server error for {}: {}", MAIL_SERVICE_CB, email, errorBody);
                                return Mono.error(new MailSendException("Mail service server error: " + errorBody));
                            })
            )
            .bodyToMono(String.class) // 성공 시 응답 본문 처리
            // --- 타임아웃 처리 ---
            // timeout 연산자를 사용하여 지정된 시간 내 응답 없을 시 예외 발생
            // MailServiceTimeoutException은 RecordException을 상속한다고 가정 (실패 기록됨)
            .timeout(DEFAULT_TIMEOUT, Mono.error(
                    () -> new MailServiceTimeoutException("Timeout after " + DEFAULT_TIMEOUT + " for " + email)) // 지연 생성
            )
            // --- 성공 로그 ---
            .doOnSuccess(response -> log.debug("[CB: {}] Successfully sent mail for {}", MAIL_SERVICE_CB, email))
            // --- 에러 매핑 및 처리 ---
            // onErrorMap을 사용하여 특정 에러를 서킷 브레이커가 기록할 예외로 변환
            .onErrorMap(this::mapToCircuitBreakerException) // 에러 매핑 메서드 참조
            // --- 블로킹 호출 ---
            // 리액티브 스트림 결과를 동기적으로 기다림
            .block();
}
public class MailServiceTimeoutException extends RecordException {

    public MailServiceTimeoutException(String message) {
        super(message);
    }

    public MailServiceTimeoutException(String message, Throwable cause) {
        super(message, cause);
    }

}

Timeout과 장애전파 방지​

적절한 타임아웃 설정은 예측하지 못한 외부 서비스나 네트워크 장애에 자원을 오래 점유당하지 않도록 해준다.

    public void callExternalApi(ExternalRequest reuqest) {
        webClient.post()
            .uri("/api")
            .contentType(MediaType.APPLICATION_JSON)
            .bodyValue(request)
            .retrieve()
            //명시적으로 5xx Status 예외도 변환
            .onStatus(status -> status.is5xxServerError(), clientResponse ->
                    clientResponse.bodyToMono(String.class)
                        .flatMap(errorBody ->
                            Mono.<Throwable>error(new ExternalServerErrorException("외부 서비스 500에러 발생"))
                        )
            )
            .bodyToMono(ExternalResponse.class)
            .onErrorMap(throwable -> {
                    if (throwable instanceof ReadTimeoutException ||
                            (throwable.getCause() != null && throwable.getCause() instanceof ReadTimeoutException)) {
                        return new ExternalServiceTimeoutException("외부 서비스 타임아웃 발생", throwable);
                    }
                    return throwable;
                })
    }

  • 5xx Status 의 경우에도 도메인 예외에 맞게 변환하여 다루어야 한다.
  • Timeout 장애 역시도 도메인 예외에 맞게 다룬다.
    • 특히 Retryable 한 경우가 많으므로 부모 예외를 잘 설정하여 복구 가능한 시스템을 만든다.
